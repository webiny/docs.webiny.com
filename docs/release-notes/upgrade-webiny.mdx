## 🔥 Precaution Measures

### Pre-production Environments First

Whenever you're doing an upgrade, we recommended that you first deploy your changes into one of your pre-production environments, like `dev` or `staging`. Once you've confirmed everything works as expected, then you can start planning moving into the production.

### Production Data Backups

When deploying to production environments, in order to prevent data loss, make sure to **backup data stored in your [OpenSearch](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/managedomains-snapshots.html) and [DynamoDB](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Backup-and-Restore.html) databases**!

If you have any additional questions or concerns, please don't hesitate to send us a message via our [community Slack channel](https://www.webiny.com/slack).

## How to Upgrade

All the guides are located in the menu on the left, for example: [6.0.0 → 6.1.0](/release-notes/6.1.0/upgrade-guide).

In Webiny v6, we have adopted a new upgrade strategy, which upgrades Webiny from your installed version to the latest (or any other of your choice) version in a single step.
This means that you don't have to upgrade Webiny version by version, but you can jump from any version to any other version, as long as the target version is higher than the currently installed one.

## Upgrade Command

Starting with `6.1.0`, we have a command `yarn webiny upgrade [version]` that will automatically upgrade your Webiny project to the specified version.
For example, if you want to upgrade to version `6.1.0`, you can run the following command:

```bash
yarn webiny upgrade 6.1.0
```

You can also specify the version as `latest`, which will upgrade your project to the latest available version, or just not send anything - same as `latest`.

If your installed version is `6.0.0`, then you need to run:

```bash
npx https://github.com/webiny/webiny-upgrades-v6 6.1.0
```

This is basically the same as the built-in command, but we did not have it available in `6.0.0`, so you need to run it directly from the GitHub [repository](https://github.com/webiny/webiny-upgrades-v6).


Once that's done, you can deploy your project.

## FAQ

### While upgrading, can I skip versions, for example from 6.0.0 to 6.10.0?

Yes, you can skip versions. The upgrade command will automatically upgrade your project to the specified version, regardless of how many versions are in between.

### Once upgraded, do we need to re-deploy the Webiny project in order to see the changes?

Yes, once you've completed the upgrade steps, in order to actually see the new features and fixes online, you need to re-deploy you project.
Which project applications you need to re-deploy? It depends on the released fixes and features, but the easiest approach would be just to deploy the entire project, using the [`yarn webiny deploy`](/cli/deploy) command.

Additionally, before you re-deploy the changes into the production environment, it's recommended that you re-deploy the changes into one of your test or preview environments first.

### Is it possible to upgrade Webiny from v5 to v6?

Differences between these two versions are so vast that it's simply impossible to do the upgrade. If you're interested in migrating your existing project to v6, feel free to reach out to us via our [community Slack channel](https://www.webiny.com/slack), and we can discuss the best approach for your project.

### How can we check what's the version of Webiny we're currently at?

In your terminal of choice and in your project root, run the following command:

```
yarn webiny --version
```


### Where can I find details on the Webiny beta releases?

You can find details about beta versions on the [create-webiny-project - npm](https://www.npmjs.com/package/create-webiny-project?activeTab=versions) page.  
If you are interested in setting up a Webiny project with a beta release, run the following command:

```
npx create-webiny-project@beta my-project
```

Additionally, set the environment variable `WEBINY_VERSION=6.0.0`. Feel free to update the tag version to your intended beta version.

Please note, we do not recommend using beta versions for production projects. However, if you want to try out features available early in a beta release, you can set up a beta version.
