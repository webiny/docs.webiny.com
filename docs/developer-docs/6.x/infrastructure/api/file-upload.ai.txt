Understanding File Upload Architecture:
- Pre-signed POST is a specific AWS S3 feature that allows temporary, limited-scope upload permissions
- The GraphQL Handler generates the pre-signed POST data, which includes:
  - URL to upload to
  - Form fields (key, policy, signature)
  - Expiration time
- The client uploads directly to S3 using a standard HTTP POST with multipart/form-data
- File metadata (size, name, type, etc.) is stored separately from the binary file itself
- DynamoDB stores structured metadata; OpenSearch enables full-text search on filenames and tags
- The two-step process (upload → metadata) means there's a brief window where the file exists in S3 but isn't "visible" in the File Manager UI

Security Considerations:
- Authentication happens on both GraphQL requests (generate pre-signed POST, store metadata)
- The pre-signed POST URL expires quickly (typically 5-15 minutes)
- S3 bucket policies only accept uploads with valid signatures
- File size limits prevent abuse
