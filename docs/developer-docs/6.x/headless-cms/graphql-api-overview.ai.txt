Read API:
- Read-only access to published content
- Designed for public-facing applications
- Queries only, no mutations
- Returns only published entries
- Optimized for performance and caching
- Safe to expose to frontend
- Requires read-only API key
- Endpoint: /cms/read/{locale}

Preview API:
- Read-only access to all content (draft, published, unpublished)
- Designed for preview features and draft mode
- Queries only, no mutations
- Returns latest version of all entries
- For authenticated editors or staging environments
- Requires preview or manage API key
- Endpoint: /cms/preview/{locale}

Manage API:
- Full CRUD access to all content
- Designed for admin panels and backend integrations
- Queries and mutations
- Access to draft, published, unpublished entries
- Supports versioning and revisions
- Never expose to frontend
- Requires full-access API key
- Can also access Preview and Read APIs
- Endpoint: /cms/manage/{locale}

Authentication:
- API keys created in Settings → Access Management → API Keys
- Passed via Authorization: Bearer {key} header
- Read API key: read-only permission, safe for frontend, Read API only
- Preview API key: preview or full access permission, server-side/authenticated only
- Manage API key: full access permission, server-side only, all APIs

API Key Scope:
- Locale scope: restrict to specific locales
- Tenant scope: tenant-specific in multi-tenant setups
- Tenant ID via x-tenant header

GraphQL Playground:
- Interactive GraphiQL interface
- Schema documentation explorer
- Query editor with autocomplete
- Add API key in HTTP Headers panel
- All three APIs (Read, Preview, Manage) have playgrounds

SDK vs Direct GraphQL:
- SDK (recommended): type-safe methods, automatic API selection, Result pattern
- Direct GraphQL: full control, complex queries, manual configuration
- Most use cases: SDK is simpler and more maintainable
