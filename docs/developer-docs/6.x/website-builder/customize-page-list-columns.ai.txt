1. No `path` prop documented — not verified to work for page data; CMS uses `values.fieldName` but pages have a flat PageDto; user's example always uses custom cell renderer, so document that pattern only
2. No `modelIds` prop — pages have no content model concept; omitted entirely
3. PageDto table included — unlike CMS (where field names come from user-defined models), page fields are fixed and worth enumerating for discoverability
4. row.data is PageDto directly — NOT row.data.values like CMS; this is the critical difference, called out clearly
5. isFolderRow guard is mandatory — page list has folders; skipping the check causes errors on folder rows
6. Default column names listed explicitly in "Position a Column" — name, createdBy, createdOn, savedOn, status, live, actions — easier than DevTools for the common case; DevTools tip still included for completeness
7. Main example uses the user's tested modifiedBy.displayName code — verified working
