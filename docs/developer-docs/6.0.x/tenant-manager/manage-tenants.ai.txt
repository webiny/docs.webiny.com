Tenant Management Workflow:
1. Create Tenant - Creates tenant record in system
   - Optional custom ID or auto-generated
   - Required: name, optional: description
2. Install Tenant - Provisions tenant resources
   - Required after creation
   - Sets up default settings and configurations
3. Enable/Disable - Access control
   - Disable: Prevents access to resources
   - Enable: Restores access

GraphQL API Patterns:
- All operations under root Mutation.tenantManager field
- All mutations return BooleanResponse type
- Error structure: { message, code, data }
- Success: data: true, error: null
- Failure: data: null, error: { message, code }

Key Operations:

1. Create Tenant
   - Input: CreateTenantInput { id?, name!, description? }
   - ID is optional (auto-generated if not provided)
   - Returns BooleanResponse

2. Install Tenant
   - Input: tenantId (required)
   - Must be called after creation
   - Provisions tenant with defaults
   - Requires admin privileges

3. Disable Tenant
   - Input: tenantId (required)
   - Prevents access to tenant resources
   - Does not delete data

4. Enable Tenant
   - Input: tenantId (required)
   - Re-enables previously disabled tenant
   - Restores access

Error Handling:
- BooleanResponse type used consistently
- Error object contains: message, code, data (JSON)
- Common error codes: TENANT_EXISTS, etc.
- Always check error field before proceeding

Complete Workflow Example:
1. Create tenant with custom ID
2. Check for creation errors
3. Install tenant (admin operation)
4. Check for installation errors
5. Tenant is ready for use
