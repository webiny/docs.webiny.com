TypeScript Types:
- Interfaces match content model field IDs
- Reference fields use CmsEntryData<T>
- Optional fields use ? suffix
- Import CmsEntryData from @webiny/sdk

Querying Operations:
- listEntries: query multiple entries
- getEntry: fetch single entry by ID
- Sort: values.{fieldId}_ASC/DESC
- Filter: where clause with operators (_eq, _gte, _contains, etc.)
- Pagination: limit and after cursor

Mutation Operations:
- createEntry: create draft entry (Manage API)
- updateEntry: modify existing entry (Manage API)
- publishEntry: make entry public (Manage API)
- deleteEntry: permanent deletion (Manage API)

Filter Operators:
- _eq, _ne: equality
- _in, _not_in: array membership
- _lt, _lte, _gt, _gte: comparison
- _contains, _not_contains: substring search

Result Pattern:
- All SDK methods return Result type
- Check result.isOk() or result.isFail()
- Access value with result.value
- Access error with result.error.message and result.error.code

Reference Fields:
- Returned as CmsEntryData<T> objects
- Contains id, entryId, values
- Values contain referenced entry fields
- Nested access: product.values.category.values.name

Server Components:
- Default in Next.js App Router
- Call SDK directly (no API route needed)
- Ideal for queries (listEntries, getEntry)
- Data fetched at build time or request time

Server Actions:
- "use server" directive
- Ideal for mutations (createEntry, updateEntry, etc.)
- Called from client components
- Return plain objects (not Result types)

Entry Structure:
- id: internal ID
- entryId: stable business ID
- createdOn, savedOn: timestamps
- createdBy, ownedBy: user references
- values: field values (your interface)
- meta: status, version, title, locked

Type Safety:
- Pass interface as generic: sdk.cms.getEntry<Product>()
- result.value.values is typed as Product
- Autocomplete and compile-time checks
- Omit generic for dynamic queries (values is any)
