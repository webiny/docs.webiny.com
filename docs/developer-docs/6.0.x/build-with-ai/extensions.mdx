## Overview

Everything in Webiny is an extension. Content models, lifecycle hooks, GraphQL schemas, Admin UI customizations, infrastructure changes, CLI commands — even the identity provider (Cognito, Auth0, Okta) is registered as an extension. This modular design means you customize Webiny by adding extensions, not by modifying core code.

All extensions share the same structure: you write the implementation in the `extensions/` folder and register it in `webiny.config.tsx`. The rest of this guide explains the different types, when to use each one, and how registration works.

## Project Structure

A Webiny project has a clean, flat layout:

- **`webiny.config.tsx`** — the single configuration file where all extensions and infrastructure settings are declared
- **`extensions/`** — the folder where all your custom code lives
- **`package.json`** — a single package file for the whole project (no monorepo)
- **`tsconfig.json`** — a single TypeScript config

You organize the `extensions/` folder however you like — by feature, by type, or by domain. The only requirement is that each extension file is referenced in `webiny.config.tsx`.

## The webiny.config.tsx File

This file exports a single React component called `Extensions`. It uses JSX to declaratively register all configuration and extensions:

```tsx webiny.config.tsx
import React from "react";
import { Api, Admin, Infra, Project } from "webiny/extensions";
import { Cognito } from "@webiny/cognito";

export const Extensions = () => {
    return (
        <>
            {/* Infrastructure */}
            <Infra.Aws.DefaultRegion name={"us-east-1"} />
            <Infra.Aws.Tags tags={{ OWNER: "me", PROJECT: "my-project" }} />

            {/* Identity provider */}
            <Cognito />

            {/* API extensions (backend) */}
            <Api.Extension src={"/extensions/ArticleModel.ts"} />
            <Api.Extension src={"/extensions/ArticlePublishHook.ts"} />

            {/* Admin extensions (frontend) */}
            <Admin.Extension src={"/extensions/AdminBranding/AdminBranding.tsx"} />

            {/* Project settings */}
            <Project.Telemetry enabled={false} />
        </>
    );
};
```

Using React components for configuration provides type safety, IDE auto-completion, and a declarative format that is easy to read and maintain.

## Extension Types

### API Extensions

API extensions customize the backend. They run inside AWS Lambda and are registered with `<Api.Extension>`. Common use cases:

- **Content models** — define models in code using `ModelFactory`
- **Lifecycle hooks** — react to events like "before entry create" or "after entry publish" using `EventHandler`
- **Custom GraphQL** — add queries, mutations, and resolvers using `GraphQLSchemaFactory`
- **Business logic** — override default behavior with custom use cases

API extension files are TypeScript (`.ts`) and follow the dependency injection pattern: a class implementing a factory interface, with dependencies declared in the constructor and exported via `createImplementation`.

```typescript extensions/ArticlePublishHook.ts
import { EntryBeforePublishEventHandler as Handler } from "webiny/api/cms/entry";
import { Logger } from "webiny/api/logger";

class OnArticleBeforePublishHandler implements Handler.Interface {
    constructor(private logger: Logger.Interface) {}

    async handle(event: Handler.Event): Promise<void> {
        if (event.modelId !== "article") {
            return;
        }
        this.logger.info("Publishing article:", event.payload.values?.title);
    }
}

export default Handler.createImplementation({
    implementation: OnArticleBeforePublishHandler,
    dependencies: [Logger]
});
```

Register in `webiny.config.tsx`:

```tsx webiny.config.tsx
<Api.Extension src={"/extensions/ArticlePublishHook.ts"} />
```

### Admin Extensions

Admin extensions customize the frontend management interface. They are standard React components registered with `<Admin.Extension>`. Common use cases:

- **White-label branding** — logos, colors, titles
- **Custom views** — new pages or modified existing ones
- **Custom components** — additional columns in content lists, custom field renderers
- **Tenant-level theming** — different branding per tenant

Admin extensions are TSX files (`.tsx`) and use React patterns you already know — hooks, context, and component composition.

```tsx extensions/AdminBranding/AdminBranding.tsx
import React from "react";
import { AdminConfig } from "webiny/admin/configs";
import logo from "./logo.png";

const { Title, Logo, Theme } = AdminConfig;

const AdminBranding = () => {
    return (
        <AdminConfig.Public>
            <Title value={"ACME Corp"} />
            <Logo
                squareLogo={<img src={logo} alt={"ACME Corp"} />}
                horizontalLogo={<img src={logo} alt={"ACME Corp"} />}
            />
            <Theme.Color palette={"primary"} color={"#6B46C1"} />
        </AdminConfig.Public>
    );
};

export default AdminBranding;
```

Register in `webiny.config.tsx`:

```tsx webiny.config.tsx
<Admin.Extension src={"/extensions/AdminBranding/AdminBranding.tsx"} />
```

### Infrastructure Extensions

Infrastructure extensions modify the AWS resources that Webiny deploys. They use Pulumi under the hood and are registered with `<Infra.Core.Pulumi>`, `<Infra.Api.Pulumi>`, or `<Infra.Admin.Pulumi>`. Common use cases:

- **Custom AWS resources** — CloudWatch alarms, extra S3 buckets, additional Lambda functions
- **Modify existing resources** — change Lambda memory, timeouts, or environment variables
- **Environment-specific infrastructure** — different configurations for dev vs prod

Infrastructure changes require a deployment to take effect — they cannot be tested with `yarn webiny watch`.

Webiny also provides declarative infrastructure components that go directly in `webiny.config.tsx` without a separate extension file:

```tsx webiny.config.tsx
<Infra.Aws.DefaultRegion name={"us-east-1"} />
<Infra.OpenSearch enabled={true} />
<Infra.Vpc enabled={true} />
<Infra.Aws.Tags tags={{ OWNER: "me" }} />
<Infra.ProductionEnvironments environments={["prod", "staging"]} />
```

### CLI Extensions

CLI extensions add custom commands to the Webiny CLI. They are registered with `<Cli.Command>` and use the `CliCommandFactory` pattern. Common use cases:

- **Data migrations** — scripts to migrate or seed content
- **Code generators** — generate boilerplate for new extensions
- **Deployment scripts** — custom deployment workflows
- **Health checks** — project-specific diagnostic commands

```tsx webiny.config.tsx
<Cli.Command src={"/extensions/MyCustomCommand.ts"} />
```

## Choosing the Right Extension Type

| You want to... | Extension type | Register with |
| --- | --- | --- |
| Define a content model in code | API | `<Api.Extension>` |
| React to content lifecycle events | API | `<Api.Extension>` |
| Add custom GraphQL queries or mutations | API | `<Api.Extension>` |
| Change the Admin logo, colors, or title | Admin | `<Admin.Extension>` |
| Add a custom column to a content list | Admin | `<Admin.Extension>` |
| Add a new page to the Admin UI | Admin | `<Admin.Extension>` |
| Add a CloudWatch alarm or custom S3 bucket | Infrastructure | `<Infra.Core.Pulumi>` |
| Change Lambda memory or timeout | Infrastructure | `<Infra.Api.Pulumi>` |
| Add a custom CLI command | CLI | `<Cli.Command>` |

## Environment-Specific Configuration

You can load extensions or configuration conditionally based on the environment:

```tsx webiny.config.tsx
<Infra.Env.Is env="prod">
    <Infra.OpenSearch enabled={true} />
    <Infra.Aws.Tags tags={{ ENV: "production" }} />
</Infra.Env.Is>

<Infra.Env.Is env={["dev", "staging"]}>
    <Infra.OpenSearch enabled={false} />
</Infra.Env.Is>
```

This is useful for enabling production-only monitoring, development-only debugging tools, or different infrastructure sizes per environment.

## Installing Pre-Built Extensions

Webiny provides official, pre-built extensions you can install with a single command:

```bash
yarn webiny extension <extension-name>
```

This downloads the extension code into `extensions/`, updates `webiny.config.tsx` to register it, and gives you full access to modify the code afterward. Pre-built extensions are a fast way to add common functionality — and since the code lives in your project, you can customize it however you need.
