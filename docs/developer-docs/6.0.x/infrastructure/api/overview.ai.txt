Understanding API Infrastructure:
- The API application is the backend for both Admin Area
- It's a serverless architecture using Lambda, API Gateway, EventBridge, SQS
- All business logic runs in Lambda functions
- API Gateway provides HTTP endpoints for GraphQL and file operations
- EventBridge enables event-driven communication between components
- SQS provides reliable message queuing for async operations

Lambda Function Types:
- GraphQL Handler: Main API endpoint, processes all GraphQL operations
- File Upload/Download: Handles file operations (pre-signed URLs, downloads)
- Background Tasks: Long-running jobs (data processing, cleanup)
- Import/Export: Bulk operations (CSV import/export)
- Event Handlers: React to EventBridge events (page publish, model changes)

API Gateway Configuration:
- HTTP API (not REST API) for lower cost and better performance
- Routes all GraphQL requests to GraphQL Handler Lambda
- Routes /upload/* and /download/* to File Manager Lambdas
- Binary media type support for file downloads
- CORS configured for Admin Area origins

EventBridge Usage:
- Emits events when entities are created, updated, or deleted
- Decouples components (enables event-driven architecture)
- Enables multiple consumers for the same event
- Event patterns allow filtering and routing

SQS Usage:
- Buffers async operations (prerendering, data processing)
- Provides retry logic and dead-letter queues
- Decouples producers from consumers
- Ensures reliable processing of background jobs

File Storage:
- S3 buckets for uploaded files and application code
- Organized by purpose (file-manager, admin-app, api-code)
- Each bucket has specific access policies and lifecycle rules

Stateful Resources (from Core):
- Amazon Cognito: User authentication and management
- Amazon DynamoDB: Primary database for structured data
- Amazon OpenSearch Service: Full-text search and analytics
- These are shown in the diagram but deployed separately

Deployment Modes:
- VPC configuration differs between development and production modes
- Lambda functions may run in private subnets (production) or public subnets (development)
- See deployment-modes documentation for network-level details
