Understanding Admin Area File Serving:
- CloudFront acts as the CDN layer, caching files at edge locations worldwide
- S3 is the origin server, storing all built files
- The browser makes many HTTP requests as it loads the SPA (HTML → JS → CSS → images)
- CloudFront cache hit/miss logic:
  - Hit: File served from edge cache (fast, no S3 request)
  - Miss: File fetched from S3, cached, then served
- Cache TTL (time-to-live) determines how long files stay cached

Static Folder and Cache Busting:
- Modern build tools (Webpack, Vite, etc.) generate content-hashed filenames
- Example: main.js → main.a1b2c3d4.js (hash changes when content changes)
- index.html references the hashed filenames
- index.html has a short cache (5 min), so users get the latest file references
- Static files have a long cache (30 days) because hash changes force new downloads
- This pattern is called "cache busting" or "immutable caching"

Caching Rules:
- /static/* → 30 days (safe because content-hashed)
- Everything else → 5 minutes (includes index.html, favicon, etc.)
- These rules can be customized in the Pulumi/IaC code

API Requests:
- Admin Area makes GraphQL requests to a different CloudFront distribution
- That distribution points to API Gateway (different infrastructure stack)
- File serving and API requests are completely separate paths
